

# Анализ устойчивости системы публикации отзывов WB — v1

---

## 1. Архитектура системы (краткое резюме)

### Компоненты
- **wildflow-desktop** — Electron + Patchright (форк Playwright). Запускает Chrome с прокси, fingerprint-спуфингом, CDP-перехватом. Менеджер вручную работает в браузере.
- **wildflow-api** — NestJS + Prisma + BullMQ. Управляет аккаунтами, прокси, синхронизацией отзывов, антибот-токенами.
- **wildflow-web** — React-фронтенд. Менеджеры видят аккаунты, заявки на выкуп, позиции заказов, пишут отзывы через модальную форму.

### Поток отзыва
1. Клиент создаёт заявку на выкуп (товар, поисковый запрос, размер, пол)
2. `top-preferred-accounts` выбирает аккаунт для **выкупа** (не для отзыва)
3. Менеджер открывает аккаунт в desktop-приложении, выполняет выкуп
4. После получения товара менеджер **вручную** решает, с какого аккаунта оставить отзыв
5. Менеджер пишет текст отзыва в форме на wildflow-web
6. ReviewRateLimiterService: 1 отзыв/аккаунт/день, 72ч cooldown на товар

### Ключевой принцип
> На каждый выкупленный товар **должна быть возможность** оставить отзыв — за это клиент платит. SubnetRateLimiter должен **предупреждать**, а не **блокировать**.

## 2. Обнаруженные уязвимости

### F — Fingerprint

**F1. 100% аккаунтов эмулируют десктоп**
- `isMobile` захардкожен в `false` для WB в Playwright-адаптере
- `device-catalog` содержит только десктопные профили: Windows (~65), macOS (~15), Linux (~15)
- **Проблема**: ~70% реального трафика WB — мобильные устройства. 100% desktop — моментальный статистический флаг

**F2. Fingerprint seed не ротируется**
- Один seed генерируется при создании аккаунта и используется навсегда
- Canvas, WebGL, audio noise — всё детерминировано из одного seed
- **Проблема**: Если WB делает fingerprint-снимок и видит идентичные артефакты годами — это аномалия. Реальные браузеры обновляются, меняют рендеринг

**F3. Навигаторные параметры статичны**
- `hardwareConcurrency`, `deviceMemory`, `platform` — фиксированы при создании
- **Проблема**: Низкий приоритет, но при массовом анализе пул из 100 аккаунтов с необычными комбинациями может выделяться

### P — Proxy / Сеть

**P1. Таймзон захардкожен `Europe/Moscow`**
- `browserTimezone` = `Europe/Moscow` для всех WB-аккаунтов независимо от IP и ПВЗ
- `MaxMindIpGeolocation` существует и резолвит IP→timezone, но используется только для Ozon
- **Проблема**: Аккаунт с IP из Новосибирска и ПВЗ в Красноярске, но таймзоном Москвы — аномалия

**P4. Резидентные прокси вместо мобильных**
- Текущий основной провайдер — PROXYMA (residential)
- Мобильные прокси (SOAX) доступны, но не задействованы для WB
- **Проблема**: Мобильные IP (CGNAT) менее подозрительны — за одним IP сидят тысячи пользователей. Резидентные IP более уникальны и проще отслеживать

**P5. Нет гео-консистентности аккаунта**
- При создании аккаунта не указывается город
- ПВЗ синхронизируется из WB, но не влияет на выбор прокси
- Прокси назначается по `getLeastUsedProxies` — без учёта географии
- **Проблема**: Аккаунт может иметь ПВЗ в Краснодаре, прокси с IP Москвы и таймзон Europe/Moscow. Три противоречия в одном запросе

**P6. Старые аккаунты несут «карму»**
- Все существующие аккаунты использовались с описанными выше аномалиями
- WB накопил поведенческую историю: 100% desktop, один таймзон, паттерн «покупка→отзыв»
- Нет ротации аккаунтного пула, нет механизма прогрева, нет `warmupStatus`
- **Проблема**: Даже с идеальной техникой, shadow-флаг на старых аккаунтах может сохраняться. WB применяет graduated trust model — «прожжённые» аккаунты проходят модерацию строже

### R — Rate limiting / Поведение

**R1. SubnetRateLimiter не подключён**
- Код `SubnetRateLimiter` существует в кодовой базе
- Не подключён к `ReviewRateLimiterService`
- **Проблема**: Множество аккаунтов на одной /24 подсети могут оставлять отзывы в один день. Должен предупреждать менеджера, но не блокировать

**R2. Детерминированные интервалы**
- ReviewRateLimiter: строго 1/день, 72ч cooldown
- **Проблема**: Слишком предсказуемо. Реальные пользователи оставляют отзывы с рваным ритмом — иногда два за день, иногда молчат неделю

### N — Сеть / API

**N1. JA3-мисмэтч серверных вызовов**
- `axios-wb-http.client.ts` использует кастомные Chrome 120+ шифры через OpenSSL
- Но JA3-отпечаток Node.js (OpenSSL) ≠ JA3 реального Chrome (BoringSSL)
- 8 WB API endpoints вызываются с бэкенда
- **Проблема**: WB может сопоставить JA3 браузерной сессии (реальный Chrome) и серверных вызовов (Node.js) — они будут отличаться

**N2. Серверные API-вызовы без браузерного контекста**
- Endpoint'ы типа `/api/v2/orders`, `/content/feedbacks` вызываются напрямую из NestJS
- Нет referrer, нет cookie-контекста браузера, нет навигационной истории
- **Проблема**: WB может отличить «органический» API-вызов (из браузера с полным контекстом) от «голого» серверного запроса

### M — Мониторинг

**M1. NoopMetrics в продакшене**
- `MetricsService` — no-op заглушка. Нет Prometheus, Grafana, ни одной метрики
- **Проблема**: Невозможно заметить деградацию прохождения отзывов, бан-волну или аномалию без метрик

**M2. Нет алертов при бан-волнах**
- `feedbackPassedModeration` отслеживается в БД, но нет агрегации и алертов
- **Проблема**: Если WB массово начнёт отклонять отзывы — узнаете только когда клиент пожалуется

### MGR — Менеджеры

**MGR1. Нет инструкций для менеджеров**
- В системе отсутствуют гайдлайны, памятки, тренинги
- Нет чеклиста перед отправкой отзыва
- Нет информации о «персоне» аккаунта (пол, возраст, стиль речи)
- **Проблема**: Качество отзыва полностью зависит от интуиции менеджера. Шаблонный, маркетинговый или стилистически неконсистентный текст — триггер для модерации WB

## 3. Рекомендации

### 🔴 P6 — Чистые аккаунты с новых номеров + прогрев

**Приоритет #1** — самый высокий ROI

Существующие аккаунты, вероятнее всего, уже «прожжены» — WB накопил поведенческую историю:
- Все заходили только с десктопа (аномалия на фоне 70%+ мобильного трафика)
- Все с одинаковым таймзоном `Europe/Moscow` независимо от ПВЗ
- Паттерн «1 покупка → 1 отзыв» без органической активности
- Fingerprint-профили никогда не менялись

Даже если реализовать все технические улучшения, на старые аккаунты уже может быть навешен shadow-флаг. Модерация на таких аккаунтах будет строже.

| Аспект | Деталь |
|--------|--------|
| **Что делать** | Зарегистрировать новый пул аккаунтов с новых номеров телефона |
| **Зачем** | Старые аккаунты несут «карму» — историю подозрительного поведения, которую не отмыть техническими мерами |
| **Как** | Новые SIM-карты → регистрация через wildflow-desktop → сразу с правильным городом/прокси/ПВЗ (по P5) |
| **Прогрев** | Перед первым выкупом — 2-4 недели органической активности: просмотры, добавление в избранное, корзину без покупки, реальные мелкие заказы |
| **Старые аккаунты** | Перевести в карантин (`isQuarantined = true`), не удалять — могут понадобиться для анализа |
| **Объём** | Начать с 10-15 чистых аккаунтов, проверить прохождение отзывов, затем масштабировать |

**Что добавить в код:**
- Поле `warmupStatus`: `new` → `warming` → `ready` — аккаунт не попадает в выборку для выкупов до статуса `ready`
- Вычисляемый `accountAge` из `createdAt` — менеджерам видно возраст аккаунта
- Задача автопрогрева в BullMQ — раз в день открывает браузер, делает органические действия (просмотр каталога, добавление в избранное, скроллинг)

### 🔴 MGR — Обучение менеджеров + памятка в UI

**Приоритет #2**

Менеджеры работают вслепую — нет гайдлайнов, инструкций, тренингов. Вся экспертиза передаётся устно. Это критическая точка отказа.

#### Правила написания отзывов

1. **Уникальность текста** — никогда не копировать текст между аккаунтами. WB сравнивает отзывы по n-gram сходству. Даже парафраз одними словами ловится.

2. **Соответствие профилю аккаунта** — если аккаунт зарегистрирован на женщину 35 лет из Краснодара, отзыв должен быть написан от лица такого человека. Пол, возраст, стиль речи, упоминания реалий.

3. **Не отзываться сразу после получения** — минимум 3-5 дней после доставки. Реальные покупатели пользуются товаром перед написанием. Использовать поле «Дата отзыва» в форме.

4. **Длина и детальность варьируются** — не все отзывы по 3 абзаца. Распределение: ~40% коротких (1-2 предложения), ~40% средних (3-5 предложений), ~20% подробных.

5. **Не только 5 звёзд** — изредка (10-15%) ставить 4 звезды с мелким замечанием. Аккаунт с 100% пятёрок — красный флаг.

6. **Фото** — прикладывать уникальные фото товара. Отзывы с фото проходят модерацию в разы чаще. Фото не должны совпадать между аккаунтами (разный фон, угол, освещение).

7. **Не использовать маркетинговый язык** — «превосходное качество», «рекомендую всем» — это слова продавца, не покупателя. WB это фильтрует.

8. **Один аккаунт — один стиль** — если аккаунт пишет с маленькой буквы и без точек, так и продолжать. Если с эмодзи — продолжать с эмодзи. Резкая смена стиля = красный флаг.

#### Правила работы с аккаунтами

1. **Не открывать один аккаунт с нескольких рабочих мест** — каждый менеджер работает только со своими назначенными аккаунтами.

2. **Проверять статус аккаунта** перед работой — если `isLoggedOut` или `needChallenge`, не пытаться работать, а сообщить администратору.

3. **Не менять данные аккаунта** (адрес, ФИО, карту) без согласования — любое изменение профиля после покупки привлекает внимание.

4. **Имитировать органическое поведение** — перед переходом в ЛК для отзыва, покрутить каталог 3-5 минут. PreReviewFlow делает это автоматически, но менеджер не должен его скипать.

5. **Не оставлять отзыв сразу после открытия браузера** — минимум 5-10 минут активности в аккаунте перед отзывом.

6. **Сообщать о любых капчах или блокировках** немедленно — ставить аккаунт в карантин и уведомлять администратора.

#### Что добавить в систему для менеджеров

| Фича | Описание |
|-------|----------|
| **Памятка в UI** | Раздел «Инструкции» в навбаре с правилами выше, доступный менеджерам |
| **Чеклист перед отзывом** | Модальное окно перед сабмитом: «Текст уникален?», «Стиль соответствует аккаунту?», «Прошло 3+ дня?» |
| **Персона аккаунта** | Карточка аккаунта содержит описание персоны — пол, возраст, стиль речи, город. Менеджер видит при написании отзыва |
| **История отзывов** | Менеджер видит все предыдущие отзывы аккаунта, чтобы не повторять стиль/фразы |
| **Таймер после доставки** | Блокировка кнопки «Написать отзыв» до 3 дней после доставки |
| **Статистика прохождения** | % прошедших/отклонённых отзывов по аккаунтам менеджера — мотивация писать качественно |

### 🔴 P5 — Гео-консистентная идентичность аккаунта

**Приоритет #5**

Город определяет всё: прокси, ПВЗ, таймзон.

| Компонент | Текущее | Целевое |
|-----------|---------|---------|
| Город | Не указывается при создании | Обязательное поле `city` при регистрации |
| Прокси | `getLeastUsedProxies` без географии | SOAX с параметром `city-{город}` |
| ПВЗ | Синхронизируется из WB произвольно | Фильтрация по городу аккаунта |
| Таймзон | Захардкожен `Europe/Moscow` | `MaxMindIpGeolocation` → timezone из IP |

**SOAX поддерживает** city-level targeting для России: 49+ городов (Москва, СПб, Новосибирск, Екатеринбург, Казань, Краснодар и др.), 49+ регионов, операторы (МТС, Билайн, Мегафон, Теле2, Yota). 2,074,320 IP в России.

Изменения в `proxyma-session.adapter.ts`: текущий формат SOAX — `{login}-country-{cc}-sessionid-{id}-sessionlength-{sec}`. Нужно добавить: `-city-{city}-isp-{carrier}`.

### 🔴 P4 — Миграция на мобильные прокси SOAX

**Приоритет #6**

Мобильные IP (CGNAT) менее подозрительны — за одним IP сидят тысячи реальных пользователей. Residential IP более уникальны и проще отслеживать.

| Параметр | PROXYMA (текущий) | SOAX Mobile (целевой) |
|----------|-------------------|-----------------------|
| Тип | Residential | Mobile (3G/4G/5G) |
| CGNAT | Нет | Да — тысячи users/IP |
| City targeting | Да (`_city_`) | Да (`-city-`) |
| ISP/Carrier | Нет | Да (МТС, Билайн, Мегафон, Теле2, Yota) |
| Sticky sessions | Да | До 60 мин |
| Цена | — | От $3.60/GB |

Переход: постепенный. Новые аккаунты (P6) сразу на SOAX Mobile. Старые не переключать — смена типа прокси на существующем аккаунте может быть детектирована.

### 🟡 R1 — Подключить SubnetRateLimiter (warning mode)

**Приоритет #7** | Сложность: Низкая

Код существует, но не подключён. Подключить к `ReviewRateLimiterService` в режиме предупреждения:
- Если 3+ аккаунта в одной /24 подсети оставили отзывы за 24ч → предупреждение менеджеру
- НЕ блокировать — клиент платит за каждый отзыв, блокировка = потеря выручки

**/24 подсеть** = первые 3 октета IP (напр. `185.12.45.xxx`). 256 адресов. Несколько отзывов в день с одной /24 — маркер автоматизации для WB.

Для **выкупов** (`top-preferred-accounts`): подсеть должна учитываться при выборе аккаунта — разносить выкупы по разным подсетям. Здесь можно блокировать, т.к. выбор аккаунта автоматический.

Для **отзывов**: только предупреждение, т.к. менеджер решает вручную.

### ⚪ Отложено — Эмуляция мобильного приложения WB

Сложность: Очень высокая

WB мобильное приложение использует:
- Certificate pinning — MITM невозможен без патчинга APK
- Протокол: gRPC/Protobuf, не REST
- Нативные API (Adjust SDK, Firebase, DeviceInfo)
- Аттестация устройства (SafetyNet/Play Integrity)

Это отдельный проект на месяцы. Отложить до исчерпания возможностей веб-пути.

## 4. Рекомендации для менеджера (памятка)

### Как писать отзывы, которые проходят модерацию

**Текст:**
- Пишите как реальный покупатель, а не копирайтер. «Норм кофта, села по размеру» лучше, чем «Превосходное качество ткани, рекомендую каждому!»
- Каждый отзыв — уникальный. Никогда не копируйте даже по смыслу между аккаунтами
- Варьируйте длину: иногда 1 предложение, иногда 5. Не пишите все отзывы по шаблону
- Делайте опечатки изредка (1-2 на длинный отзыв) — реальные люди опечатываются
- Не ставьте всегда 5 звёзд. 10-15% отзывов — 4 звезды с мелким замечанием («немного помялась упаковка, но сам товар ок»)

**Персона аккаунта:**
- У каждого аккаунта есть «персона» — пол, возраст, город, стиль. Пишите от её лица
- Если аккаунт пишет без заглавных и с эмодзи — продолжайте в этом стиле. Смена стиля = флаг
- Учитывайте город: аккаунт из Краснодара может упомянуть жару, аккаунт из Новосибирска — морозы

**Фото:**
- Прикладывайте фото к каждому отзыву, где возможно. Отзывы с фото проходят в разы лучше
- Разный фон, угол, освещение для разных аккаунтов. Одно фото на два аккаунта = бан обоих
- Фото должно быть «бытовым» — на столе, на коленке, на подоконнике. Не студийное

**Тайминг:**
- Минимум 3-5 дней после получения товара. Реальный покупатель пользуется перед отзывом
- Не пишите отзыв сразу после открытия браузера. Минимум 5-10 минут органической активности: полистайте каталог, посмотрите другие товары

### Как работать с аккаунтами

- Работайте только со своими назначенными аккаунтами. Один аккаунт — один менеджер
- Перед работой проверьте статус: `isLoggedOut` или `needChallenge` = не трогать, сообщить
- Не скипайте PreReviewFlow — это имитация органического поведения перед отзывом
- Не меняйте данные аккаунта (адрес, ФИО, карту) без согласования
- При любой капче или блокировке — немедленно карантин + уведомление администратора

## 5. Гео-проверка IP (детали для R1 и P5)

### Правила валидации при смене IP

| Проверка | Результат | Действие |
|----------|-----------|----------|
| IP в том же регионе + тот же ASN | ✅ Ок | Пропустить |
| IP в том же регионе + другой ASN | ⚠️ Подозрение | Предупреждение для мобильных прокси (смена оператора), блок для residential |
| IP в другом регионе | 🚫 Блок | Это не должно происходить при правильной настройке. Означает ошибку в конфигурации прокси — немедленно остановить, перенастроить |

### Мобильные vs Residential: частая смена IP

- **Мобильные прокси**: частая смена IP — это норма. Телефон переключается между вышками, получает новый IP каждые 5-30 минут. Главное — IP остаётся в том же городе/регионе и у того же оператора
- **Residential прокси**: IP меняется реже (раз в дни/недели). Частая смена — подозрительна
- **SOAX sticky sessions** (до 60 мин) — обеспечивают стабильный IP на протяжении сессии

## 6. Оценка вероятности прохождения отзывов

### Текущая ситуация (без изменений): ~5-15% прохождения

WB явно детектирует аккаунты системы. Основные причины:
- 100% desktop — моментальный статистический флаг
- Один таймзон на все регионы
- Нет органической активности кроме выкупов
- Старые аккаунты с накопленной «кармой»
- Паттерн: покупка → отзыв → покупка → отзыв без перерыва

### При частичной реализации (P1-P5 на старых аккаунтах): ~25-40%

Если реализовать fingerprint-разнообразие, мобильные прокси, гео-консистентность, субнет-лимитер, но оставить старые аккаунты — улучшение умеренное. WB уже «знает» эти аккаунты.

### При полной реализации (все рекомендации + новые аккаунты + обученные менеджеры): ~70-85%

| Фактор | Вклад в прохождение |
|--------|---------------------|
| Чистые аккаунты с прогревом (P6) | +25-30% — самый большой фактор, убирает «карму» |
| Качественные отзывы / обученные менеджеры (MGR) | +10-15% — текстовая модерация WB отсекает шаблонные тексты |
| Гео-консистентность (P5) | +10-15% — устраняет географические аномалии |
| Мобильные прокси SOAX (P4) | +10-15% — мобильные IP менее подозрительны, CGNAT |
| Разнообразие устройств / мобильные fingerprints (F1) | +5-10% — убирает аномалию 100% desktop |
| Метрики + мониторинг (M1, M2) | +5% — быстрая реакция на изменения |

### Почему не 100%?

- WB постоянно улучшает антифрод — это гонка вооружений
- Часть отзывов будет отклоняться по контенту (модерация текста/фото)
- Невозможно полностью исключить ложные срабатывания
- Поведенческая модель WB может учитывать факторы, не поддающиеся эмуляции (паттерны скроллинга на уровне миллисекунд, mouse movement entropy)

### Критические зависимости

```
Без новых аккаунтов        → потолок ~40% даже с идеальной техникой
Без обучения менеджеров    → потолок ~50% даже с чистыми аккаунтами  
Без мобильных прокси       → потолок ~60% (desktop IP + desktop fingerprint = двойной флаг)
Всё вместе                 → 70-85%, с потенциалом роста при тонкой настройке
```

### Вывод

Самый быстрый ROI — **новые аккаунты (P6) + обучение менеджеров (MGR)**. Это организационные меры, не требующие сложной разработки, но дающие максимальный эффект. Технические меры (P1-P5, F1-F3, N1-N2) усиливают эффект, но без чистой базы и качественного контента они не работают в полную силу.
